# Spec 写作指南

## 使用场景

### 读者

Spec 同时写给两类读者：

- **人（你）** — 需求 review、方案取舍、半年后回看"为什么这么做"
- **AI agent（Claude 等）** — 基于 spec 生成 plan、按 plan 落地执行、跨 session 接手时快速对齐设计

两类读者都依赖 spec 做"决策证据归档 + 架构骨架锁定"，所以本指南的约束对两侧都成立。

### 触发流程

```
需求（对话 / 想法）
  └→ 你 review / brainstorm，锁定方向
       └→ AI 按本指南生成 spec（docs/superpowers/specs/）
            └→ 你 review spec，确认决策 / 调整边界
                 └→ AI 生成 plan（docs/superpowers/plans/）
                      └→ 如需补细节，回头把 spec 里的占位替换为 plan 链接
                           └→ AI 执行 plan、提交 commits
```

### 何时写 spec

- 多文件 / 架构性改动
- 有取舍的决策（> 1 个合理方案）
- 跨 PR / 跨 session 的工作（需要锁决策证据给下个 session）
- 需要独立验收（验收判据不能省）

### 何时不写

- Trivial 改动（几行 bug fix、一次性脚本、纯 typo / rename）
- 决策被上层规范锁定且无取舍（直接 plan 或直接改）
- 探索性 prototype（先跑通再回头补 spec）

### Spec 的主要作用

1. **锁定"为什么"** — 决策证据归档，防止半年后自己或下一个 agent 重走同一坑
2. **锁定架构骨架** — 不同 plan 落地都必须落在同一设计上；AI 换 session 也不会跑偏
3. **立验收标杆** — 做完 vs 没做完有明确判据，不靠感觉

## Spec 与 Plan 的边界

**Spec = why / what / 轻量 how + 验收**
**Plan = 顺序 / 文件清单 / 自动化 gate / 具体命令**

### 判据（写的时候自问）

| 这条内容... | 归 spec | 归 plan |
|---|---|---|
| 不同实施者都需要知道才能理解设计 | ✅ | |
| 只有具体落地时才用得上（改哪个文件、跑哪条命令） | | ✅ |
| 删掉之后论证不成立 | ✅ | |
| 删掉之后只是少个例子，结论还站得住 | | ✅ |
| DB migration / API rename 一改就过期 | | ✅ |
| 锁死了架构方向，后续 plan 必须遵守 | ✅ | |

### "轻量 how" 的颗粒度

允许写进 spec：

- ✅ 目录树（说明模块切分）
- ✅ 3–5 行 sketch / pseudocode（说明形状）
- ✅ 关键字段名（解释不变量时）
- ✅ CLI 形状（`cmd <arg> [--flag]`）
- ✅ 文件名 / 路径（说明模块归属、分层）

不写进 spec（属 plan）：

- ❌ 完整 zod schema / SQL CREATE TABLE / tsconfig JSON
- ❌ 详尽 before→after 文件变更表
- ❌ 函数签名枚举、所有 CLI flag
- ❌ 完整实现片段
- ❌ 正文里用 `curl` / `pnpm xxx` / `grep` 等具体命令解释 how

判据复述：**删掉论证不成立 → 留 spec；删掉只是少个例子 → 去 plan**。

### Plan 占位机制

spec 早于 plan 生成时，不确定的细节不强行编，用固定占位：

```
> TODO(plan): 字段清单与 migration 细节
```

plan 落地后回头改成：

```
> 字段清单见 plan §3.2
```

**格式固定**：前缀 `>`（blockquote）让占位视觉突出；`TODO(plan)` 可 grep，AI 扫 spec 时立刻发现"这里有未填细节"。

**禁止** `<!-- plan: ... -->` HTML 注释做占位 —— 渲染后看不见，容易漏。

## 一、内容边界

### 必须写

- 目标、非目标、前提
- 关键产品 / 架构决策及其动机
- **高层 how**（按上面"轻量 how"颗粒度）：
  - 系统分几块、各块职责与边界
  - 核心流程的骨架（时序、状态迁移、失败分支）
  - 协议 / 契约的形态（鉴权模型、错误分类、版本策略）
  - 数据模型的形状（实体、关系、生命周期、关键不变量）
- **方案对比与 tradeoff**：候选方案、拒绝理由、保留的代价
- 验收（feature 级 + Phase 级，见 §六）
- 风险、已知缺口、未决问题

### 不要写

- 函数签名、完整伪代码 —— 代码
- 字段级所有列定义、HTTP header 字面量 —— plan 或代码
- 正文里用具体命令行解释 how（`curl` / `pnpm xxx` / `grep`）—— plan 或脚本
  - 例外：验收章节"自动化侧"列 gate 命令清单是允许的（命令本身就是判据）
- 本机绝对路径（`~/code/...`、`/Users/...`）
- Commit message、具体工具调用

### 颗粒度判据（例）

- 写"设备鉴权用服务端签发的长期不透明 token、只存 hash" —— **对**
- 写"`devices.token_hash TEXT UNIQUE NOT NULL`、`/api/sync/push` 读 `Authorization: Bearer ...`、status 401 则回 onboarding" —— **越权，属 plan**

文件路径 / 目录名 / 模块名 **不禁止**。能帮读者锁定"这个决策落在哪一层"就该写，别把它们当禁忌词。

## 二、结构

### 标准骨架

```
# <特性名> Design
<frontmatter，2~4 行>

## 0. 概览
  0.1 目标
  0.2 非目标
  0.3 前提 / 起点
  0.4 关键决策一览

## 1. 方案对比 (Alternatives considered)
  候选方案 → 拒绝理由 → 采纳方案的代价

## 2. 架构
  模块切分 / 职责 / 数据流骨架

## 3. 数据模型
  实体形状、关系、生命周期、关键不变量

## 4. 协议 / 契约
  鉴权模型、错误分类、版本策略（形态，非字段）

## 5. 关键流程
  时序 / 状态机 / 失败分支（图 + 文字）

## 6. Phase 切分（如有）
  每个 Phase 同深度 what / why / 轻量 how

## 7. 验收
  Feature 级 + 每 Phase 级（见 §六）

## 8. 风险与未决
```

章节顺序不是死规，但"方案对比"和"架构"必须在正文前半段，不能藏到附录。

### Frontmatter 模板

```markdown
**Status**: Design / In Progress / Done
**Date**: YYYY-MM-DD
**Scope**: <一句话：改什么、不改什么>
**Plan**: <plan 文件路径，如已生成>
```

格式统一的 frontmatter 让 AI 扫一遍 specs 目录时能直接按状态 / 日期筛。

## 三、方案对比与 tradeoff

这是 spec 最容易偷懒的部分。决策不是凭空蹦出来的，读者（包括半年后的自己）需要知道"我们评估过什么"。

### 每个关键决策需要回答

1. **候选方案**：至少列 2 个现实可选项（含"不做"这个选项，如果适用）
2. **评估维度**：复杂度 / 延迟 / 成本 / 可运维 / 一致性 / 扩展性 —— 挑本次真正重要的 3~5 个
3. **拒绝理由**：为什么其他方案不合适，用一句话收口
4. **保留的代价**：采纳方案的已知短板是什么、什么时候会痛

### 推荐格式

```
### 3.2 同步协议：LWW vs CRDT vs OT

候选：
- A. Last-Write-Wins + 服务端 op log
- B. CRDT（Automerge / Yjs）
- C. OT（OT.js 等）

维度与评估：
| 维度        | A (LWW) | B (CRDT) | C (OT) |
|-------------|---------|----------|--------|
| 实现复杂度  | 低      | 高       | 高     |
| 并发语义    | 单字段精度，够用 | 字段内合并 | 字符精度 |
| 离线时长    | 任意    | 任意     | 受限   |
| 运维门槛    | 低      | 中高（库版本锁） | 高  |

采纳：A。理由：无实时协作需求，单字段 LWW 足够覆盖 99% 场景；
B/C 的合并优势在我们的工作区 / 标签场景里价值低，复杂度不成比例。

代价：
- 两端同时改同一字段，后到的全量覆盖，可能丢上下文
- 字段级 LWW 依赖时钟，需靠 opId 兜底
```

### 何时可以省略

- 决策是业界标准默认值（"用 HTTPS"）—— 不用写
- 决策被别处（架构规范、技术雷达、上游 spec）锁定 —— 引用即可
- 项目内同决策已在更早 spec 对比过 —— 引用上次即可

**剩下所有"有替代方案"的决策**都要写对比。决策表里每条都该能指到正文的对比章节。

## 四、决策表写法

决策表是速查入口，不是决策本身。

### 规则

1. 一条一行，**保持速查属性** —— 不塞段落、文件路径清单、实现细节
2. 背后理由 / tradeoff 写在正文，不塞单元格
3. 按类型分表：产品决策 / 架构决策 / 实现约束
4. 编号稳定，正文用 `决策 #7` 引

### 反例

```
| 17 | Protocol version | Semver PROTOCOL_VERSION from packages/protocol/
package.json.version. /api/health exposes a compatibility window
(minSupportedProtocolVersion, ...). Extension distinguishes hard-block
from soft-warning |
```

塞了三段话 + 文件路径 + 字段名 —— 决策表塌了。

### 正例

```
| 17 | 协议版本策略 | Semver + 服务端声明兼容窗口；区分硬阻断与软告警 |
```

字段和文件路径留给 plan / 代码；tradeoff（为什么不用"强制一致版本"或"不做版本检查"）在正文对比章节展开。

## 五、数据模型与协议的写法

只写 **形状与职责**，不写列定义和 header 名。

- 每个实体：存在理由、所有者、生命周期、与其他实体的关系、关键不变量
- 协议：鉴权模型（不透明 token / JWT / session）、错误分类（哪些可重试 / 哪些终态）、版本策略形态

不写：`id text PK`、`x-opentab-protocol-version` 这种字面。

> 字段级的细节一旦写进 spec，文档会随 DB migration 不停漂，且和 code-as-truth 打架。

实体关系超过三张表 → 画 ER 图。关键流程 → 画时序图或状态机图。一图胜 200 行文字。

## 六、验收（Acceptance）

Spec 必须写验收。没有验收的 spec 只是愿望。

### 三要素（缺一不可）

1. **验收标准（what）** —— 做到什么算"这个 feature 完成"。外部可观测
2. **如何验收（how）** —— 用什么手段证明标准达到。黑盒剧本 / 指标 / 自动化 gate / 人工 checklist
3. **通过条件（pass bar）** —— 清晰判定线：N 个场景全过、指标达到阈值、某清单全 tick

### 必须写到的层级

- **Feature 级**：整个 spec 交付算完成的总验收
- **Phase 级**（如 spec 分 Phase）：每个 Phase 独立验收，phase done 判据清晰
- **关键流程级**（可选）：失败风险高的流程单独给验收

### 两侧验收（推荐模式）

对有用户交付面的 feature，把验收切成两边：

- **自动化侧（AI / CI 跑）** —— 列 gate 命令清单：typecheck / test / lint / build / 项目专用 check。命令本身就是判据，这里写命令是允许的
- **人工侧（你验）** —— 视觉 smoke / copy reality check / 操作语义 / 运行后观察 —— 用用户视角语言，不写命令

两侧不可替代：自动化抓结构性破损，人工抓 copy、视觉、操作直觉。

### 无用户视角的 feature（库 / 基础设施）

用"外部可观测"替换"用户视角"：

- 上游消费者看到什么行为
- 边界契约（API / 类型 / 错误码）是否满足
- 文档 / 类型提示是否让调用方一次用对

### 格式建议

```
### 7.1 Phase 1 验收

验收标准：
- 用户完成设置向导后，扩展可离线继续工作，恢复网络后数据收敛到服务端
- 两台设备在线时，同一实体的后到改动覆盖先到改动
- 服务端注销某设备后，该设备下一次请求失败并回到向导入口

如何验收（自动化侧）：
- `pnpm -w typecheck` / `pnpm -w test` / `pnpm -w lint` / `pnpm -w build` 全绿
- `pnpm check:protocol-version` 报告兼容窗口一致

如何验收（人工侧 / Scenario）：
- Scenario A（单设备引导上行）：新用户完成向导 + 本地数据上传
- Scenario B（双设备收敛）：两台设备分别改同一工作区名，5 秒内两端收敛到后到值
- Scenario C（吊销）：Web 端吊销设备，下次同步应触发重新授权

通过条件：
- 四条命令 0 errors
- 三个 scenario 全过，证据存档
- 无新增崩溃 / 数据丢失
```

### 写法纪律

- Scenario 用用户视角，不塞 `curl` / `grep` 做场景描述
  - ✅ "第二台设备选择 Download，两台设备应看到一致的工作区树"
  - ❌ "`curl -H 'x-protocol-version: 1.0' ...` 返回 `applied: [...]`"（用命令当 scenario 描述）
- 自动化侧 gate 命令清单允许具体命令
  - ✅ `` `pnpm -w typecheck` 全绿 ``
  - ❌ "跑一下相关测试保证 pass"（模糊到不可判）

## 七、术语与格式纪律

- 每个核心术语在 §0 定义一次，全文只用这一个写法
- schema 命名风格（camelCase / snake_case 分界）说明一次
- 引用用稳定编号，避免"如上所述"
- 删修饰词 —— 英文："strictly / globally / comprehensive / purely"；中文："严格地 / 全面地 / 彻底地 / 统一地 / 完全地"。它们多数时候不增加信息

---

## 八、长度与节奏

**没有硬线，读不顺就拆**：

- 章节开始有"堆砌"感 → 独立成 Phase 或子章节
- 决策表条数多到需要反复滚动对照正文 → 按类型分表
- 整份 spec 读到一半已经忘了开头 → 拆子 spec 或加 §0.4"关键决策一览"做索引
- Phase 之间颗粒度要可比 —— Phase 2 只能写出 15 行 outline，要么补到同深度、要么独立成后续 spec

唯一的硬要求：**每个章节 / Phase 开头 2~3 行 TL;DR**，让读者（和 AI）能快速决定要不要读下去。

---

## 九、不要做的事

- ❌ 决策只给结果不给对比（"We use X" 然后没了）
- ❌ 把 plan 的 TODO 步骤清单塞进验收（验收写"做到什么算对"，不是"做了哪几步"）
- ❌ 同一件事在决策表、正文、Appendix 反复强调三次
- ❌ Non-goals 和 Backlog 列同一条
- ❌ 用"see implementation"糊弄 how 的核心思路
- ❌ 方案对比写成"A 不行，选 B" —— 必须说 A 为什么不行
- ❌ 用 HTML 注释做 plan 占位（`<!-- plan: ... -->`）—— 渲染后看不见
- ❌ 正文用具体命令解释 how（`curl ...` / `pnpm xxx ...`）—— 验收自动化侧除外

---

## 十、落笔前自查

1. 每个关键决策是否都有"方案对比 + 拒绝理由 + 保留代价"？
2. How 的核心思路是否写到位（读者能画出架构图 / 时序图 / 目录树）？
3. 有没有越权写完整 schema / 函数签名枚举 / 所有 flag 清单 / 绝对路径？
4. Phase 之间颗粒度是否可比？
5. 同一术语是否全文一个写法？
6. 决策表 / 正文 / Appendix 有没有在讲同一件事三遍？
7. Non-goals 和 Phase X backlog 有没有重复？
8. 验收三要素（标准、手段、通过条件）是否齐全？feature 级和每个 Phase 级都有吗？
9. 未确定的细节是否用了 `> TODO(plan): ...` 固定占位，而不是"看着办"？
