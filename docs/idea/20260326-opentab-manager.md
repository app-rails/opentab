## Problem Statement

Chrome 用户面临三个核心痛点：
1. **标签页太多找不到** — 开 20+ tab 后，favicon 都看不清，找目标页面靠猜
2. **想按项目/场景分组管理** — 同时进行多个项目（工作、学习、side project），tab 混在一起无法切换上下文
3. **关掉后想找回来** — 关闭的 tab 消失了，Chrome 历史记录搜索体验差，经常找不到"上周看的那个页面"

现有方案的不足：
- **Workona** 验证了 workspace 概念可行，但收费且闭源，用户无法自部署或审计数据
- **OneTab / Session Buddy** 只做保存/恢复，无 workspace 概念
- **AI Tab Manager 们（Tabaroo、ATO 等）** 只做 AI 分组，缺乏持久化 workspace 管理
- **开源方案（Tab Manager Plus、Tabli）** 功能停留在"列出所有 tab"层面，远未达到 workspace 级别

## Premises

> These are the foundational assumptions. If any premise is wrong, rethink everything below.

1. **开源透明是核心差异化** — 闭源商业产品占据高端体验位，而开源方案功能薄弱。开源 + 高质量体验的组合存在空白。
2. **手动 workspace 先行，AI 后加** — 稳定可控的手动 workspace 切换是基础。AI 自动分组是体验加速器，不作为 V1 核心依赖。
3. **Chrome 应用商店是主发布渠道** — 用户通过商店发现和安装，GitHub 作为源码主页和社区阵地。
4. **账号必做（默认匿名访客）** — 使用 better-auth 实现认证，默认创建匿名访客账号实现零摩擦进入。为后续云同步、跨设备、付费升级做准备。
5. **全页面管理界面优于 Popup** — Tab 管理是重操作场景，需要足够的空间展示三栏信息（workspace 列表 / 管理的 tabs / 浏览器 live tabs），Popup 太局促。

## What Makes This Cool

**"开源版 Workona + Tab 时光机"**

- **Workspace 一键切换** — 点击"工作"，只看到工作相关的 tab；点击"Side Project"，瞬间切换上下文。真正的 workspace 隔离和持久化。
- **Tab 时光机** — 关掉的 tab 自动归档到 workspace，可搜索可恢复。不是浏览器历史记录，是带 workspace 上下文的智能归档。
- **三栏 Dashboard** — 左栏 workspace 切换，中栏管理的 tab（含归档），右栏 live tabs。一目了然，拖拽操作。
- **开源 + 自部署** — 数据完全在用户手里。
- **匿名开箱即用** — 安装后 0 秒可用，不弹注册框。

## Landscape Context

**Layer 1 — 大家都知道的：**
Tab 管理是浏览器生产力领域的长青需求。Chrome 原生 Tab Groups 能力有限（不持久化、不支持 workspace 切换），给第三方扩展留下巨大空间。

**Layer 2 — 当前趋势：**
2025-2026 年 AI tab 管理成为热点，但大多数只做"一键 AI 分组"单一功能。Workona 和 Side Space 提供最完整的 workspace 体验，但都是闭源商业产品。

**Layer 3 — 我们的判断：**
市场两极之间存在空白：功能单一的开源工具 vs 体验完整但闭源收费的商业产品。OpenTab 做"体验接近 Workona 但完全开源、本地优先、支持自部署"的产品。

## Architecture Overview（概念级）

```
┌─────────────────────────────────────────────────────────┐
│                    Chrome Browser                        │
│                                                         │
│  ┌─────────────┐    ┌──────────────────────────────┐    │
│  │  Background  │    │     Full Page (tabs.html)     │    │
│  │  Service     │◄──►│  ┌────┬──────────┬────────┐  │    │
│  │  Worker      │    │  │ WS │ Managed  │  Live  │  │    │
│  │             │    │  │List│  Tabs    │  Tabs  │  │    │
│  │ - Tab 事件监听│    │  │    │(含归档)  │(浏览器) │  │    │
│  │ - Chrome API │    │  │    │          │        │  │    │
│  │ - 消息路由   │    │  └────┴──────────┴────────┘  │    │
│  └──────┬──────┘    └──────────────────────────────┘    │
│         │                        │                       │
└─────────┼────────────────────────┼───────────────────────┘
          │                        │
          │         Zustand        │
          │      ┌─────────┐       │
          └──────┤ Dexie.js├───────┘
                 │(IndexedDB)│
                 └─────┬─────┘
                       │ (Phase 4: 可选同步)
                 ┌─────▼─────┐
                 │  Hono API  │
                 │ better-auth│
                 │ (轻量后端)  │
                 └───────────┘
```

**关键数据流：**
1. **Tab 创建/关闭** → Background SW 监听 `chrome.tabs.onCreated/onRemoved` → 写入 Dexie → 通知全页面更新
2. **Workspace 切换** → 全页面发消息给 Background → Background 通过 workspace 切换机制（见下方 Tech Spike）更新浏览器 tab 可见性 → 更新 Dexie
3. **Tab 归档恢复** → 全页面从 Dexie 读取归档 tab → 用户点击 → Background 调用 `chrome.tabs.create`
4. **搜索** → 全页面直接查询 Dexie（title + URL 索引）

**MV3 Service Worker 生命周期管理：**

MV3 的 Background Service Worker 会在 ~30 秒无活动后被终止。这是核心架构风险，必须正确处理：

- **事件监听不会丢失** — Chrome 保证在 SW 被终止后，如果有注册的事件（如 `chrome.tabs.onCreated`）触发，会重新唤醒 SW 并执行 listener。所以 tab 事件不会丢失。
- **内存状态会丢失** — SW 重启后内存中的变量会被清空。所有状态必须持久化到 Dexie/Chrome Storage，SW 唤醒后从存储恢复。
- **Zustand 状态在全页面中** — 核心 UI 状态由全页面的 Zustand store 管理，不依赖 SW 内存。SW 只做事件中转和 Chrome API 调用。
- **心跳/keepalive 不需要** — 不要用 `chrome.alarms` 或 keepalive hack 来保持 SW 存活，这违反 MV3 设计理念且可能导致商店审核问题。

**离线 / 后端不可用策略：**

- 扩展的核心功能（workspace 管理、tab 归档、搜索）100% 本地运行，不依赖后端
- 后端仅用于账号认证和可选同步
- 如果后端不可用：扩展安装时生成本地 UUID 作为临时 accountId，存入 Chrome Storage。当后端恢复可用时，后台自动重试创建匿名账号并关联本地数据
- 用户无感知——后端是否在线不影响任何本地功能

## Data Model（概念级）

```
Account（服务端权威，本地缓存）
├── id: string (UUID)
│   └── 本地优先：安装时立即生成本地 UUID，后端可用时注册并关联
├── type: 'anonymous' | 'registered'
├── email?: string
├── serverSynced: boolean (是否已与后端同步)
├── createdAt: Date
└── updatedAt: Date

Workspace
├── id: string (UUID)
├── accountId: string → Account.id
├── name: string (e.g., "工作", "Side Project")
├── color: string (hex, 用于视觉区分)
├── icon?: string (emoji 或 icon name)
├── position: string (fractional indexing，如 "a0", "a1", "a0V"，使用 fractional-indexing 库)
├── createdAt: Date
└── updatedAt: Date
    注：默认 workspace 通过 Settings.defaultWorkspaceId 标识，不在 Workspace 上冗余

TabCollection（集合：一组 tab 的逻辑分组，可一键全部打开）
├── id: string (UUID)
├── workspaceId: string → Workspace.id
├── name: string (e.g., "早会", "开发", "Snapshot 2026-03-26 14:30")
├── position: string (fractional indexing)
├── isDefault: boolean (每个 workspace 有一个默认集合，新 tab 自动归入)
├── createdAt: Date
└── updatedAt: Date

CollectionTab（集合内的 tab 记录）
├── id: string (UUID)
├── collectionId: string → TabCollection.id
├── url: string
├── title: string
├── favicon?: string
├── chromeTabId?: number (当前打开时有值，未打开时为空)
├── position: string (fractional indexing)
├── addedAt: Date
└── lastAccessedAt: Date

Settings（单例，存储在 Chrome Storage sync 中，支持跨设备同步）
├── newTabOverride: boolean (是否替代新标签页，默认 false)
├── tabGroupsEnabled: boolean (是否启用 TabGroups 模式，默认 false，Phase 3 引入)
├── defaultWorkspaceId: string (唯一的"默认 workspace"标识，不可删除)
├── theme: 'light' | 'dark' | 'system'
└── shortcuts: Record<string, string>
```

**Dexie Schema Versioning：**
- 使用 Dexie 内置的版本管理（`db.version(N).stores({...}).upgrade(fn)`）
- 每次 schema 变更递增版本号，编写 upgrade 函数迁移数据
- 扩展更新时 Dexie 自动执行迁移，用户无感知

**关键关系：**
- Account → 多个 Workspace
- Workspace → 多个 TabCollection（含一个 default 集合）
- TabCollection → 多个 CollectionTab
- **一个 tab 只属于一个 Collection**（V1 简化设计。用户可通过"复制到其他集合"实现跨集合共享）
- 每个 Workspace 自动有一个"未分组"默认集合（isDefault=true），新 tab 自动归入
- Tab 快照功能 = 创建一个新 TabCollection，将当前所有 live tab 的 URL/title 写入

### Phase 1: Workspace 核心

**目标：** 用户可以创建/管理 workspace，将 tab 分配到 workspace，一键切换上下文。

**⚠️ Phase 1 使用方案 C（纯 Dashboard 管理）：workspace 切换仅更新中栏显示内容，不操作浏览器 tab 可见性。TabGroups 集成在 Phase 3 作为 opt-in 功能加入。**

**业务逻辑：**
- Workspace CRUD（创建、重命名、改颜色/图标、排序、删除）
- 删除 workspace 时，其中的 active tab 不关闭，回到"Default" workspace
- Workspace 切换 = 更新中栏显示的 tab 列表（Phase 1 不操作浏览器 tab 可见性）
- "Default" workspace 不可删除，捕获所有未分配的 tab

**数据流：**
- Tab 打开 → Background 监听 `chrome.tabs.onCreated` → 分配到当前活跃 workspace → 写入 Dexie
- Tab URL 变化 → Background 监听 `chrome.tabs.onUpdated` → 更新 Dexie 中的 title/url/favicon
- Tab 在 workspace 间移动 → 更新 WorkspaceTab.workspaceId

**UI 任务：**
- [ ] 左栏：Workspace 列表，支持新建、选中高亮、右键菜单（重命名/改色/删除）
- [ ] 中栏：当前 workspace 的 active tab 列表，显示 favicon + title + URL
- [ ] 右栏：浏览器所有 live tab 列表
- [ ] 拖拽：右栏 tab 拖到中栏 → 加入 workspace
- [ ] 中栏 tab 拖拽排序

**Chrome API 关键点：**
- `chrome.tabs.query` — 获取所有打开的 tab
- `chrome.tabs.onCreated / onRemoved / onUpdated` — 监听 tab 生命周期
- `chrome.tabGroups` — 可选，用 tab group 实现 workspace 隔离的视觉效果
- `chrome.tabs.update({ active: true })` — 激活 tab
- 注意：Manifest V3 没有 `chrome.tabs.hide/show`，需要用 `chrome.tabGroups` 或 window 管理来模拟 workspace 切换

**Milestone 验收标准：**
- ✅ 可创建、重命名、删除 workspace（Default 除外）
- ✅ 右栏显示浏览器所有 live tab
- ✅ 可将 live tab 拖拽加入 workspace
- ✅ 切换 workspace 时，中栏内容正确更新
- ✅ 新开的 tab 自动归入当前活跃 workspace
- ✅ 刷新/重启扩展后数据不丢失

### Phase 2: Tab 时光机

**目标：** 关闭的 tab 自动归档到 workspace，可搜索可恢复。这是核心差异化功能。

**业务逻辑：**
- Tab 关闭 → Background 监听 `chrome.tabs.onRemoved` → WorkspaceTab.status 从 `active` 改为 `archived`，记录 archivedAt
- 归档 tab 保留在 workspace 中，灰色/半透明显示
- 点击归档 tab → `chrome.tabs.create({ url })` → status 恢复为 `active`
- 搜索范围：当前 workspace（workspace 内搜索）或全局（全局搜索栏）
- 搜索字段：title + URL，使用 Dexie.js 索引实现

**数据层：**
- Dexie 索引：WorkspaceTab 表对 `title` 和 `url` 建立复合索引
- 归档保留策略（V1 简单处理）：保留全部，后续可加 TTL 或数量限制

**UI 任务：**
- [ ] 中栏分区显示：上方 active tabs，下方 archived tabs（灰色/半透明）
- [ ] Archived tab 鼠标悬停显示"点击恢复"
- [ ] 中栏搜索框：过滤当前 workspace 的 tab（含归档）
- [ ] 顶部全局搜索栏：跨 workspace 搜索，结果按 workspace 分组显示
- [ ] 搜索结果高亮匹配文字

**Milestone 验收标准：**
- ✅ 关闭 tab 后，中栏自动出现对应的灰色归档条目
- ✅ 点击归档 tab 可恢复打开
- ✅ Workspace 内搜索可找到历史 tab
- ✅ 全局搜索可跨 workspace 找到历史 tab
- ✅ 归档数据在扩展重启后持久存在

### Phase 3: 体验打磨 + 发布

**目标：** 从"能用"到"好用"，达到 Chrome Web Store 发布标准。

**体验优化：**
- [ ] 快捷键系统：Cmd/Ctrl+Shift+1~9 切换前 9 个 workspace
- [ ] 新标签页替代模式：Settings 中可开启，将全页面设为新标签页
- [ ] 主题支持：light/dark/system 自动跟随
- [ ] 空状态设计：新 workspace 的引导提示
- [ ] 状态栏：底部显示当前 workspace 信息摘要
- [ ] 动画过渡：workspace 切换、tab 归档/恢复的微动画
- [ ] 批量操作（stretch goal）：多选 tab → 批量移动到其他 workspace / 批量关闭 / 批量恢复
- [ ] TabGroups 模式（opt-in）：Settings 中可开启，启用后 workspace 切换会通过 `chrome.tabGroups` collapse/expand 控制 tab 可见性。首次启用时自动为现有 workspace 创建对应 TabGroup

**健壮性：**
- [ ] Tab ID 对账机制：`chrome.runtime.onStartup` 时，全量对比 Dexie 中 active 状态的 chromeTabId 与 `chrome.tabs.query` 结果。Chrome 重启后 tab ID 会变，需通过 URL 匹配重新关联或将失联 tab 标记为 archived
- [ ] 扩展更新/崩溃恢复：基于上述对账机制自动修正
- [ ] 数据一致性：定期对比 Dexie 数据与实际 Chrome tab 状态

### Approach A: 极简 MVP — "开源 Workona 平替"
Summary: 纯 Chrome 扩展，workspace 切换 + tab 持久化 + 关闭历史，最小技术栈

## Recommended Approach

**Approach B: 全栈架构 + 三栏全页面 Dashboard**

选择理由：账号体系 day 1 就有比后补成本低。历史搜索直接解决"关掉后想找回来"的高频痛点。三栏全页面提供足够空间展示所有信息，避免 Popup 的局促和 Side Panel 的 API 限制。架构预留 AI 接口，后续扩展不需重构。

## Tech Spike: Workspace 切换实现方式

Manifest V3 **没有** `chrome.tabs.hide/show` API。这是核心功能的实现约束，需要在 Phase 0 末期做技术验证。

### 方案 A: TabGroups collapse/expand

```
切换到 Workspace "工作":
1. 将"工作" workspace 的 active tabs 设为一个 TabGroup，expand
2. 将其他 workspace 的 TabGroup 全部 collapse
```

- **优点：** 实现简单，API 稳定
- **缺点：** collapsed tab group 仍然显示在 tab 栏（只是折叠了），不是真正的"隔离"。用户仍能看到其他 workspace 的折叠标签。
- **用户体验：** 适合 tab 数量中等（每个 workspace 5-15 tab）的用户。大量 tab 时 tab 栏仍然拥挤。

### 方案 B: 多窗口隔离

```
切换到 Workspace "工作":
1. 如果"工作" workspace 有关联窗口，focus 该窗口
2. 如果没有，创建新窗口并将"工作"的 active tabs 移入
3. 可选：最小化其他 workspace 窗口
```

- **优点：** 真正的视觉隔离，每个 workspace 一个干净的窗口
- **缺点：** 多窗口管理复杂；窗口间移动 tab 有 API 限制；用户习惯不同
- **用户体验：** 最接近 Workona 的体验，但需要用户接受多窗口工作流

### 方案 C: 纯 Dashboard 管理（不控制浏览器 tab 可见性）

```
切换 workspace 只更新 Dashboard 中栏的显示内容
不主动隐藏/显示浏览器中的 tab
用户通过 Dashboard 点击 tab 来激活/聚焦
```

- **优点：** 最简单，无 Chrome API 限制
- **缺点：** 没有真正的"上下文切换"感觉，只是一个增强版书签管理器
- **用户体验：** 门槛最低，但"workspace 切换"的核心卖点被削弱

### 推荐

**Phase 1 先用方案 C（纯 Dashboard），Phase 3 升级到方案 A（TabGroups）。**

理由：方案 C 让我们最快验证 workspace 管理 + tab 归档的核心价值，不被 Chrome API 限制阻塞。方案 A 作为 Phase 3 体验打磨的一部分加入，此时核心功能已稳定。方案 B 作为进阶选项保留。

## Open Questions

1. **归档保留策略** — V1 保留全部归档 tab，但长期使用后数据量可能很大。是否需要 TTL？用户手动清理？Phase 2 末期决定。
2. **License 选择** — MIT（宽松，利于传播）vs AGPL（要求衍生品开源，保护社区）。Phase 4 前决定。
3. **Tab 跨 Window 管理** — 用户可能有多个 Chrome 窗口，workspace 是否跨窗口？V1 限定当前窗口，后续扩展。
4. **拖拽库选择** — dnd-kit（轻量、现代、React 优先）vs @hello-pangea/dnd（react-beautiful-dnd 的维护 fork）。Phase 1 开始时决定。
5. **导出格式** — workspace 右键菜单中的"导出"具体是什么格式？JSON？Markdown 链接列表？Phase 3 决定。
6. **同步冲突策略** — Phase 4 的"last-write-wins"在 workspace 重命名、tab 排序等场景可能不够。已知限制：两台设备同时重命名同一 workspace 时会丢失一方修改。Phase 4 启动时评估冲突频率，last-write-wins 可能已足够。
7. **归档去重策略** — 同一 URL 反复打开关闭会产生多条归档记录。V1 默认每次关闭 = 独立归档条目（保留完整时间线）。后续可考虑按 URL 合并并递增 visitCount。
8. **Chrome API 错误处理** — `chrome.tabs.create/query` 等 API 可能失败（tab 数量限制、无效 URL 等）。统一策略：UI 层通过 toast 通知用户，非关键操作静默重试 1 次，关键操作（workspace 切换）失败时回退到上一状态。
